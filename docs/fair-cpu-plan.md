## 0. 決定事項と背景

最強 CPU 計画の実施（2026-07-08）で、**出荷されている challenger の攻撃評価が相手の実手札を
参照している**ことが判明した（同計画 §9 の「非公開情報の参照禁止」原則と実装が矛盾していた）。
ユーザーは **B案 = 公平性原則を貫く**を選択した:

- CPU の評価は**公開情報のみ**を入力とする。これは絶対制約であり、以後どのフェーズでも緩めない
- 公平化により CPU は一時的に弱くなる（覗き見版との直接対決で 43.9% を実測済み）。
  これは強化ではなく**原則の修正**なので、55% 採用ゲートは適用しない。弱体幅は記録のみ行う
- 公平化後の CPU を新たな基準（fair-gen001）として、**公平な土俵の上で**改めて最強を目指す。
  ここからは 55% ゲートが復活する（対戦相手は公平チャンピオンプール）

思考時間制約（1 秒未満）・決定性制約（同一シード再現）は前計画から引き継ぐ。

## 1. 公平性の定義（何が公開情報か）

評価関数・行動選択が参照してよいもの:

- 自分側の全情報（手札の中身を含む）
- 相手の場・トラッシュ・メモリ・遺物・ライフ・チャージ
- 相手の**手札枚数**・**山札枚数**（枚数は公開。中身と順序は非公開）
- 公開デッキリスト（プリセットデッキの構成）
- 観測履歴（相手がプレイ・防御・トラッシュ等で公開したカード）

参照してはいけないもの: 相手の手札の**中身**、山札の**順序と中身**。

境界の注意: `chooseAiDefense` が**防御側自身の選択**として自分の手札を読むのは合法。
違反になるのは、**攻撃側の評価**が相手の `chooseAiDefense` を実手札つきで呼んで予測に使う場合。

## 2. フェーズ構成

```
F0: 公平化           ← 覗き見の棚卸しと公開情報ベースへの置換（ゲートなし・原則修正）
F1: 再ベースライン    ← fair-gen001 凍結、全指標の取り直し
F2a: 重み再探索       ← 公平な土俵で 28+ 重みを再チューニング（55% ゲート復活）
F2b: カードカウンティング精緻化 ← 公開情報推定器そのものを強くする
F2c: ターン内プランニング再設計 ← 前計画 Phase 3 の失敗原因を潰して再挑戦
F2d: 相手ターン読み   ← 任意。F2b の推定手札分布を応手予測に使う
```

F2a が有望な理由: 現行 28 重みは**覗き見つき予測を前提に**チューニングされた値。
予測器が公開情報ベースに変わると最適な重みバランスも変わるはずで、公平化で失った分の
一部を取り返せる見込みが高い。

## 3. F0 — 公平化

**作業**:

1. **棚卸し**: 評価・行動選択の全経路から相手手札/山札の非公開参照を列挙し、合法/違反に分類する。
   既知の違反箇所（網羅は棚卸しで確認。行番号は 2026-07-08 時点）:
   - `chooseAttackEvaluationDefense` の既定フォールバック（`src/game.ts:2642` → `chooseAiDefense(defender, ...)` 経由で実手札参照、`:2598` で中身の価値も読む）
   - STRIKE 評価の `chooseStrikeHandDefense(opponent, ...)`（`:2511-2513`）
   - `bestDamagingAttacker`（`:2156`。相手の実手札で防御可否を判定。`:2200` の呼び出し元がどのプロファイルかも確認）
   - beginner / classic 手筋の経路も同様に確認する
   - 実施結果（2026-07-08）:
     - 違反: `chooseAttackEvaluationDefense` の覗き見フォールバック、STRIKE 評価内の `chooseStrikeHandDefense(opponent, ...)`、classic 優先行動の `bestDamagingAttacker`、beginner の攻撃可否での `legalHandDefenders(defender, ...)`
     - 合法: 実際の防御解決で防御側自身が `chooseAiDefense` / `chooseStrikeHandDefense` から自分の手札を読む経路、相手の手札枚数・山札枚数・公開盤面/トラッシュ/遺物を読む評価
2. **置換**: 攻撃側の相手モデルを `estimatePublicHandDefenseValue`（前計画 Phase 3.5 で実装済み）
   ベースに一本化する。`publicHandDefenseWeight` の「候補フラグ」ではなく、公平版を既定挙動にする
3. **ガードテスト**: 「公開情報と手札枚数が同一なら、相手の実手札の中身が違っても
   `chooseAiAction` の選択が同一」を単体テストで固定する（推定器単体の既存テストを行動選択レベルへ拡張）。
   以後の全変更に対する回帰ガードとして常設する
4. **弱体幅の記録**: 公平版 vs 旧覗き見版（gen001）の直接対決を 2 シードで計測し、
   数値を balance-history に記録する（**ゲートではない**。事実の記録のみ）

**完了条件**: ガードテスト green、`npm run check` green、チュートリアル非波及
（`npx vitest run src/game/tutorial.test.ts`）。

## 4. F1 — 再ベースライン

challenger は計測器なので、挙動が変わったら全基準値を取り直す（前計画 §8-5 と同じ定型）:

1. 公平版を `docs/assets/ai-champions/fair/fair-gen001.json` として凍結。以後のチャンピオンプールは
   fair 系列で運用する（旧 gen001 は参考スパーリング相手として残してよいが、ゲート判定には使わない）
2. 6 デッキリーグ（2 シード、league_report.py で合否）
3. 盛り上がり指標（excitement_metrics.py）
4. ストレスデッキ回帰（`npm run balance:cost`）
5. apex 再探索（`npm run tune:apex`）
6. beginner の強さ較正確認（対 challenger 勝率 5-20% 基準が維持されているか）
7. balance-history 先頭に「CPU 公平化・再ベースライン」エントリを追記し、
   **旧（覗き見版）基準の勝率数値との比較禁止**を明記する（Python 廃止時と同じ作法）

ここでリーグ合否（単色 45-55% / 先攻 48-52%）が崩れたら、それは「覗き見が隠していた
バランス問題」であり、カード/ルール側の調整課題として起票する。

## 5. F2 — 公平な土俵での再強化（55% ゲート復活。対戦相手は fair プール）

- **F2a 重み再探索**: `npm run tune:ai` を fair プール適応度で実行。前計画で整備した
  複数パス・エリート継承・変異幅縮小をそのまま使う。公平化で評価地形が変わっているため、
  前計画 Phase 1 の「据え置き」結果は参考にならない（再探索の価値がある）
- **F2b カードカウンティング精緻化**: 推定器を「デッキリスト − 観測済みカード」の
  真のカウンティングに進化させる。観測履歴の追跡、攻撃ごとのブロック確率化
  （単一期待値ではなく、この攻撃が手札防御で止まる確率の見積り）など。1 改善 = 1 実験
- **F2c ターン内プランニング再設計**: 前計画 Phase 3 は幅 2/3 で 35%/29.3% と大幅弱化したが、
  これは「1 手スコアの足し算」という系列評価の設計不備が原因の可能性が高い。着手時はまず
  **異常の原因調査**から入る（幅 3 の平均思考時間 1.4ms が幅 1 の 2.1ms より短い計測結果は
  試合が異常に短い＝自滅している疑い）。その上でターン終了盤面の専用評価
  （残アクション・次ターン被害リスクを含む）を設計し直す
- **F2d 相手ターン読み（任意）**: F2b の推定手札分布を応手候補の列挙に使う。
  F2c までの思考時間実測に余裕がある場合のみ

## 6. やらないこと（前計画 §9 から引き継ぎ・更新）

- 非公開情報の参照（本計画の存在理由。ガードテストで機械的に禁止する）
- ニューラルネット評価関数（当面対象外のまま）
- 検証専用 AI と出荷 CPU の乖離
- ルール・カード側の変更をこの計画に混ぜること（別コミット・別検証）
