## 0. 目的と位置づけ

Break Duel のバランス検証パイプライン（リーグ・盛り上がり指標・ストレスデッキ回帰・apex 探索）は
すべて challenger AI を「計測器」として使っている。計測器が弱いと、

- 強すぎるカード/コンボを AI が使いこなせず、壊れが**見逃される**
- AI の下手さ由来の偏りが、カード/ルールの問題として**誤検出される**

の 2 種類の誤りが起きる。したがって challenger の強化は、今後のカード追加の方向性判断と
ルールのバランス調整の**信頼性を底上げする基盤投資**である。

副次的な利点:

1. apex デッキ探索（`npm run tune:apex`）の探索品質が上がる（弱い AI では最強デッキ候補の序列が歪む）
2. 盛り上がり指標（逆転率・スイング）の妥当性が上がる（下手なプレイ由来の「偽の逆転」が減る）
3. プレイヤー向け最高難度としてそのまま提供できる
4. 新カードの「上手く使われたときの上限」を実装前に測れる
5. beginner AI の強さ校正（勝率 5-20% 基準）の基準点が安定する

### 制約（「最強」の定義）

- **無制限の最強ではない**。以下の 2 制約下での最強を目指す:
  1. **思考時間制約**: ブラウザで 1 ターンの思考が体感即応（目安 1 秒未満）であること
  2. **決定性制約**: 同一シードで再現可能であること（A/B 検証とビット一致確認の前提）
- 実装は TypeScript の単一実装（`src/game.ts`。2026-07-08 に Python シミュレータを廃止し統一）。
  ブラウザとシミュレーション CLI が同じ AI コードを共有するため、「シミュレーション専用 AI」と
  「出荷される CPU」が乖離する構造は存在しない。バランス合否は常に実際に出荷される CPU で判定される

## 1. 現状アーキテクチャ（2026-07-08 時点）

- challenger は **28 項目の採用済み線形重み + 既定オフの実験重み（CHALLENGER_WEIGHTS）による 1 手評価**
  - `src/game.ts`（重み `CHALLENGER_WEIGHTS`）、`scoreAiAction()` / `boardAiScore()`
- 自ターン内でもアクションを **1 手ずつ貪欲に選択**（3 アクションのターン全体は計画しない）
- 相手の防御選択のみ防御予測で読む。相手ターン全体は読まない
- 重み探索は `npm run tune:ai`（`scripts/tuneAiProfiles.ts`、ランダム突然変異、適応度 = 対 baseline 直接対決勝率 + 0.15×床値）
- 採用基準の前例: 旧重みとの直接対決で **55% 以上勝ち越し**（WP3 では候補が超えられず据え置き）

## 2. 改善の全体戦略

強化の見込みが大きい順ではなく、**リスクと実装コストが低い順**に段階を踏む。
各フェーズは独立に採用/却下でき、どこで止めても資産（計測基盤・記録）が残る。

```
Phase 0: 計測基盤（ガントレット）      ← まずこれがないと以降の主張が全部弱い
Phase 1: 重み空間の徹底探索            ← コード変更なし・低リスク
Phase 2: 評価関数の項目拡張            ← 線形のまま知識を足す
Phase 3: 自ターン内プランニング        ← 最大の伸びしろ（貪欲 → ターン全体最適化）
Phase 3.5: 相手手札推論（カードカウンティング） ← 低コスト・高効果。Phase 4 の読みの土台（§5 末尾）
Phase 4: 相手ターン読み（2手読み）     ← 高コスト。Phase 3.5 の結果を見て判断
```

自己対戦 NN 学習は思考時間制約・説明可能性の点で当面**対象外**（線形重みの勾配学習までは Phase 1 の拡張として可）。

## 3. Phase 0 — 計測基盤: チャンピオンプール & ガントレット

**目的**: 「強くなった」の判定を、単一 baseline との直接対決から**歴代チャンピオンのプール**との
総当たりに格上げする。単一相手への勝ち越しは「その相手へのメタ」（じゃんけんの後出し）に
過学習しやすく、世代を重ねると強さが循環する恐れがあるため。

**作業**:

1. `scripts/aiGauntlet.ts` を新設（`scripts/tuneAiProfiles.ts` の対戦部を再利用）
   - 入力: 候補の weights JSON（または挙動フラグ）、チャンピオンプール（過去採用版の weights のリスト）
   - 対戦: プール各員と 7 デッキ（break/control/fire/water/wind/earth/apex）× 先後 × N 試合
   - 出力: 対プール全体勝率、対戦相手ごとの勝率、デッキごとの床値（最低勝率）、シードつき JSON
2. チャンピオンプールの初期化: 現行 CHALLENGER_WEIGHTS を第 1 世代として
   `docs/assets/ai-champions/gen001.json`（仮）に凍結保存
3. 挙動変更（Phase 2 以降）に備え、weights だけでなく「AI バージョン」を切り替えられる形にするかは
   Phase 2 着手時に判断（それまでは weights JSON で足りる）

**完了条件**: 現行 challenger を自分自身と戦わせて勝率 ≈50%（先後補正込み）になることをもって
ハーネスの健全性を確認。`npm run check` green。

## 4. Phase 1 — 重み空間の徹底探索（コード変更なし）

**目的**: 現行 28 重みの探索を計算資源で押し切る。実装リスクゼロで取れる分を先に取る。

**作業**:

1. `scripts/tuneAiProfiles.ts` の適応度を「対 baseline」から「対チャンピオンプール」に差し替え可能にする
2. 探索の強化（いずれも自然言語ループで回しやすい単純な拡張）:
   - 反復数・候補数・games_per_seat の増量（前回: iterations 24 / 12 games 規模）
   - エリート保存つき反復（優秀候補を次世代のベースにして再変異）
   - 変異幅の段階縮小（粗く探索 → 上位候補の近傍を細かく）
3. **複数シードでの最終確認**: 最終候補は 2 シード以上・十分な試合数で現行と直接対決

**採用基準**（以降のフェーズも共通）:

- 対現行チャンピオン直接対決 **55% 以上**（2 シード以上で一貫）
- 対プール全体で勝ち越し、かつ床値（最弱デッキ勝率）が現行より悪化しない
- 誤差圏内なら**据え置き**（WP2/WP3 と同じ判断基準）

**期待値の注記**: WP3 で一度探索済みのため伸びは小さい可能性が高い。ここでの主目的は
「線形重みの上限に近いこと」を計算量をかけて確認し、Phase 2 以降の投資判断の根拠にすること。

## 5. Phase 2 — 評価関数の項目拡張（線形のまま知識を足す）

**目的**: 重みの調整ではなく、評価に**見えていないもの**を見えるようにする。

**候補となる新特徴**（各項目 = 1 実験。1 つずつ入れてガントレットで検証）:

| 候補 | 現状の盲点 |
| --- | --- |
| メモリ効果の個別評価 | 30 種超の memory effect が一律 `memory=51` ベースで簡略化されている疑い |
| チャージ計画性 | チャージの価値が「今アクションが増えるか」中心で、次ターン以降の資源計画がない |
| ライフレース推定 | 「あと何ターンで殴り切れるか vs 殴り切られるか」の明示的な比較がない |
| 山札残量・デッキ切れ距離 | リソース決着（数 % 残す設計）への意識が弱い可能性 |
| デッキタイプ条件つき重み | 全デッキ共通の固定重み。自デッキのアーキタイプ別に重みセットを切り替える案 |

**計測の多様性について（優先順位の指針）**: 強化の順番は「①エンジン全体の底上げ（Phase 1-4）→
②デッキ別重み（上記候補の最終行）→ ③同一デッキ内の性格違い（攻め型/守り型の重み摂動プロファイル）」。
③は計測器の偏りに対する保険で、`damage`/`bad_attack`/`hand_card` あたりを摂動した重みセットを
リーグに混ぜると「最適プレイでは壊れないが特定の打ち方で壊れるカード」を検出できる。
①②が済むまでは着手しない（エンジンが強くなるほど③の必要性は下がる）。

**初期仮説リスト（AI 実装乖離の監査より）**:

Python シミュレータ廃止（2026-07-08）前の監査で、旧 Python 実装と TS 実装の challenger に **14 項目の実装差**が
見つかった。TS がゲーム本体でありスペック準拠のため TS を正としたが、以下は**旧 Python 側にだけあった評価知識**であり、
ガントレット検証つきで TS challenger への導入を試す価値がある。Phase 2 の初期仮説として扱う:

- カード価値ボーナス: `draws_on_play` +8 / `filters_on_play` +10 / `opponent_draws_on_play` −12 /
  `draws_on_successful_defense` +8
- `PLAY_AI` の場圧迫ペナルティ −12（場が埋まるほど召喚の評価を下げる）
- pierce（防御貫通）攻撃の手札防御素通り評価（貫通攻撃は手札防御で止まらない前提で採点する）
- classic `bestChargeFuel` の `min(3, ...)` ハードコード修正相当（チャージ燃料選択の上限処理）
- STRIKE 候補のタイブレーク規則

全 14 項目の監査記録は**付録 A**に完全収録してある（監査の一次記録はこの文書のみ。元の Python コードは削除済み）。

**作法**:

- 1 特徴 = `src/game.ts` への実装 + 単体テスト + ガントレット、を 1 サイクルとする
- 特徴を追加したら Phase 1 の探索を新重みを含めて再実行（新特徴の重み 0 = 現行と等価、から出発
  すると安全。重み 0 で現行とリーグ結果が**ビット一致**することを先に確認する）
- 不採用の特徴も balance-history に記録（再提案防止）

### Phase 3.5 — 相手手札推論（カードカウンティング）

Phase 2 の作法で回す追加特徴だが、規模が大きいため独立フェーズとして扱う。
**推奨タイミングは Phase 3 完了後・Phase 4 の着手判断の前**（Phase 4 の 2 手読みは
相手の応手予測の精度に依存するため、先にこちらを入れる方が投資判断が正確になる）。

**狙い**: challenger は既に「相手の手札防御まで予測してダメージが通るかを判定する」ロジックを
持つが（付録 A #4）、相手手札の中身は分からない前提の推測に留まる。ここに観測情報からの
推定を足すと、攻撃を通す/防御燃料を残すという勝敗直結の判断の精度が**既存パイプライン全体で**上がる。

**設計方針**:

- 対戦デッキはプリセット（break/control/fire/water/wind/earth/apex）でリストが実質公開情報のため、
  「デッキリスト − 観測済みカード（プレイ・防御・トラッシュ等で公開されたもの）＝ 残り候補」の
  **引き算（カードカウンティング）**で足りる。確率的な当て推量は不要
- 相手の手札枚数（公開情報）と残り候補から期待分布を計算し、既存の防御予測・
  「ダメージが通る攻撃」判定の入力に使う
- リスト非公開の対戦形態が将来増えた場合は「観測済みカードの引き算のみ」の弱い版に自動縮退する形にする
- 推定は**観測済み公開情報のみの純粋関数**として実装する（ゲーム状態の非公開部分への参照を持たない）。
  決定性制約を壊さず、探索を伴わないため思考時間への影響もほぼゼロ

**遵守事項と検証**:

- §9 の「非公開情報の参照禁止」との区別を単体テストで保証する（入力が観測履歴のみであること、
  相手手札・山札順オブジェクトに触れないこと）
- Phase 2 の作法どおり: 推論オフ（フラグ or 重み 0）で現行とリーグ結果がビット一致することを先に確認し、
  ガントレット 55% 基準で採用判定する

## 6. Phase 3 — 自ターン内プランニング（本命）

**目的**: 「1 アクションずつ貪欲」を「ターン全体（3〜4 アクション）の系列を評価して最良系列を選ぶ」に
変える。現行最大の構造的弱点であり、伸びしろが最も大きいと見込む。
（例: 貪欲では「先にチャージしてから展開」より「今一番点の高い手」を選んでしまう）

**設計方針**:

- 自ターン内のアクション系列を**ビームサーチ**（幅 K・深さ = 残アクション数）で列挙し、
  ターン終了時点の `boardAiScore()` + 系列中の攻撃価値で評価
- 乱数が絡む効果（ドロー等）は、その時点で系列を打ち切って現行評価に落とす（決定性の維持）
- ビーム幅と評価回数に上限を置き、ブラウザでの思考時間を計測してから幅を決める
- 実装は `src/game.ts` に直接行い、ガントレットで効果を確認してから採用判定する

**リスクと対策**:

- 系列評価のコア関数を小さく保ち、貪欲評価との等価条件（ビーム幅 1 = 現行）をテストで固定する
- チュートリアル（`src/tutorial.ts`）は固定進行のため、challenger の挙動変更が波及しないか
  `npx vitest run src/game/tutorial.test.ts` を必ず回す

## 7. Phase 4 — 相手ターン読み（任意・保留可）

Phase 3 採用後の challenger に対して、相手の返しターン（最善応手 1 手番）まで読む 2 手読みを試す。
計算量が跳ね上がるため、Phase 3 の思考時間実測に余裕がある場合のみ着手。
効果が誤差圏内なら据え置きで完了とする。
Phase 3.5 の手札推論が採用済みなら、相手の応手候補の列挙にその推定手札を使う
（採用されていない場合、2 手読みの応手予測は精度が出にくい前提で期待値を下げて判断する）。

## 8. 各フェーズ共通の検証・記録手順（1 イテレーションの定型）

自然言語の改善ループはこの定型を 1 周として回す:

1. **仮説**: 何を変えると何が強くなるはずか、を 1 行で書く
2. **実装**: `src/game.ts` に変更。既存挙動を変えないはずの部分は同シードリーグの
   ビット一致で確認
3. **ガントレット**: 対チャンピオンプール勝率・床値を 2 シード以上で計測
4. **採用判定**: 55% 基準 + 床値悪化なし。誤差圏内は据え置き
5. **採用時の再ベースライン**（重要）: challenger は計測器なので、採用したら
   - 6 デッキリーグ（2 シード、`league_report.py` で合否）
   - 盛り上がり指標（`excitement_metrics.py`）
   - ストレスデッキ回帰（`npm run balance:cost`）
   - apex 再探索（大きく強くなった場合）
   を**すべて再実行**し、新しい基準数値を balance-history に記録する。
   ここで既存デッキのバランスが崩れて見えたら、それは「弱い AI が隠していた問題」であり、
   カード/ルール側の調整課題として起票する（← 本計画の本来の目的そのもの）
6. **記録**: `docs/balance-history.md` 先頭にエントリ追記（採用/却下とも）。採用時は
   チャンピオンプールに新世代の weights/バージョンを追加
7. **最終ゲート**: `npm run check` green

## 9. やらないこと

- ニューラルネット評価関数（思考時間・説明可能性の難点。線形の限界が実証されたら再検討）
- 出荷される CPU と別物の検証専用 AI（シミュレーションと実プレイの乖離を生むため禁止。
  TS 単一実装により構造的にも作れないが、方針としても禁止を維持する）
- CPU だけが使える非公開情報（相手手札・山札順）の参照 — 公平性の原則に反する。
  ただし**観測済みの公開情報**（プレイ・防御・トラッシュで見えたカード、公開デッキリスト、手札枚数）
  **からの推論は参照にあたらず合法**（人間プレイヤーにもできるカードカウンティング。Phase 3.5 参照）
- ルール・カード側の変更をこの計画に混ぜること（計測器の変更とゲームの変更は必ず別コミット・別検証）

## 10. コマンドリファレンス（2026-07-08 時点・すべて TS 実装）

シミュレーションは TypeScript 単一実装（約 300 試合/秒。旧 Python 比 6〜10 倍速）。

```bash
# 一括チェック（typecheck + vitest 全 280 テスト + build。ストレスデッキのガードレール込み）
npm run check
```
