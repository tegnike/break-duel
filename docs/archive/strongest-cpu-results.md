# 最強 CPU 計画 実施結果まとめ

作成日: 2026-07-08
対象: challenger CPU 強化計画（`docs/archive/strongest-cpu-plan.md`）
結論: **計測基盤は採用。出荷 CPU の既定挙動は据え置き。**

## 要約

今回の目的は、Break Duel のバランス検証で計測器として使う `challenger` CPU を、ブラウザでも即応できる決定的な範囲で強化できるか確認することだった。

結果として、ガントレット基盤とチャンピオンプールは有用だったため採用した。一方で、重み探索・評価関数拡張・ターン内プランニング・公開情報ベースの手札推定はいずれも採用基準の **55% 以上勝ち越し**に届かず、または明確に弱体化した。そのため、実際にプレイヤーが対戦する challenger の既定挙動は変更していない。

## 採用したもの

### AI ガントレット

追加:

- `scripts/aiGauntlet.ts`
- `npm run gauntlet:ai`
- `docs/assets/ai-champions/gen001.json`

役割:

- 現行 challenger を `gen001` として凍結保存する
- 候補重み JSON を、チャンピオンプール相手にミラーデッキ先後ペアで評価する
- 評価対象は `break / control / fire / water / wind / earth / apex`
- 出力は pool win rate、デッキ別勝率、床値、対チャンピオン別勝率

健全性確認:

| 条件 | 結果 |
| --- | ---: |
| 現行 challenger vs gen001 | 50.0% |
| 全デッキ別勝率 | 50.0% |

同一シードで先後を入れ替えるペア方式にしたため、自己対戦は構造的に 50% へ揃う。以後の候補比較で、先後・シード由来のノイズを減らせる。

### 重み探索の拡張

`scripts/tuneAiProfiles.ts` を以下に対応させた。

- チャンピオンプール適応度
- 複数パス探索
- エリート継承
- 変異幅の段階縮小
- 候補 JSON / チャンピオン JSON の `weights` フィールド読み込み

これは今後のCPU強化候補を回すための基盤として残す。

## 採用しなかったもの

### Phase 1: 重み空間探索

探索条件:

- `iterations 36`
- `passes 3`
- `elite-count 4`
- `games-per-seat 16`
- seed `730001`

探索中の小標本 best は pool 56.1% まで出たが、独立確認では採用基準に届かなかった。

| 候補 | seed | games/seat | pool win rate |
| --- | ---: | ---: | ---: |
| top1 | 910001 | 120 | 51.5% |
| top1 | 920001 | 120 | 51.8% |

判断: **据え置き**。2シードで一貫して55%以上に届かなかった。

### Phase 3: 自ターン内プランニング

ビームサーチを実装し、`turnPlanBeamWidth` で有効化できるようにした。既定値は `1` で、現行貪欲選択と等価。

| 幅 | seed | games/seat | pool win rate | 判断 |
| ---: | ---: | ---: | ---: | --- |
| 1 | 540001 | 20 | 50.0% | 現行等価 |
| 2 | 940001 | 100 | 35.0% | 不採用 |
| 3 | 940001 | 100 | 29.3% | 不採用 |

思考時間の簡易測定:

| 幅 | 平均 | 最大 |
| ---: | ---: | ---: |
| 1 | 2.1ms | 53.6ms |
| 2 | 8.0ms | 773.9ms |
| 3 | 1.4ms | 35.6ms |

幅2でも1秒未満には収まったが、勝率が大きく落ちたため採用しない。

### Phase 3.5: 公開情報ベースの手札推定

実装:

- `estimatePublicHandDefenseValue()`
- `publicHandDefenseWeight`
- 実手札の中身ではなく、公開デッキリスト・公開ゾーン・手札枚数から手札防御期待値を推定
- unit test で「実手札の中身が違っても公開情報と手札枚数が同じなら推定値が同じ」ことを固定

検証:

| 条件 | seed | games/seat | pool win rate |
| --- | ---: | ---: | ---: |
| `publicHandDefenseWeight=1` | 950001 | 120 | 43.9% |

判断: **不採用**。既存の実手札参照ベースの評価より大きく弱い。

### Phase 4: 相手ターン読み

未着手で完了扱い。

理由:

- Phase 3 のターン内プランニングが不採用
- Phase 3.5 の手札推定も不採用
- 返しターン読みは前提となる応手予測精度が不足しており、計算量だけ増える可能性が高い

## 今回分かったこと

1. 現行 challenger の線形評価は、単純な重み探索では大きく上回りにくい。
2. 旧 Python 側にあった評価知識をTS側へ戻しても、単独では採用ラインに届かなかった。
3. 自ターン内ビームサーチは、現行の1手評価をそのまま積み上げるだけだと弱くなる。ターン終了評価や乱数効果の扱いを根本から設計し直す必要がある。
4. 公開情報ベースのカードカウンティングは公平だが、現在の採点式では実手札を使う評価より弱く、単純置換には向かない。
5. 一方で、候補を安全に比較するガントレット基盤は整った。今後のCPU強化は「候補を作る」より先に、このガントレットを標準ゲートとして使える。

## 今後やるなら

今回の結果から、次にCPUを強くするなら以下のどれかが現実的。

1. **ターン内プランニングの評価式を作り直す**
   - 1手スコアの足し算ではなく、ターン終了盤面・残アクション・次ターンリスクを専用に評価する。
2. **候補行動の枝刈りを改善する**
   - ドロー・サーチ・チャージなど、状態が変わる行動を正しく扱う。
3. **デッキ別CPUを本格的に設計する**
   - 今回の簡易バイアスは弱かった。やるならデッキごとに採用候補行動と評価式を分ける必要がある。
4. **完全情報シミュレーションではなく、実プレイ用の公平AIとして再設計する**
   - 現在の検証AIは相手手札を評価に使える。出荷CPUの公平性を重視するなら、公開情報ベースに寄せた別設計が必要。
