## 1. 背景（まず読むこと）

### 1.1 何が行われたか

このゲームは以前から勝率バランス（45-55%）は良好でしたが、試合展開が平坦という問題がありました。原因はデータで特定されています。

- ダメージが常に 1 点固定で、1 試合の最大ライフスイングが 2 点止まり
- 先制ダメージが平均 8 手番目（試合の前半 4 割が無風）
- 先に 2 点差をつけた側の勝率 79%（中盤で実質決着）
- 切札 power 4 が実質召喚不可能（コスト 4 > 最大 3 アクション）

これを解決するため、2026-07-03 にルールを全面改訂しました。詳細は `docs/game-spec.md`（正仕様）と `docs/balance-history.md`（採用判断と検証数値）を参照してください。改訂の柱は次の 7 点です。

1. 突破ダメージ制（ダメージ = 攻撃召喚獣の power。1〜4 点。2026-07-04 改定: 当初は攻撃値の段階制だったが「パワーと削りを一致させる」方針でダメージ = power に変更。攻撃値補正は防御突破・討伐判定専用）
2. 初期ライフ 5→8
3. 基本アクション 2→3（チャージで最大 4。コスト=power は維持、power 4 はチャージ経由で 1 ターン素出し可能）
4. ブレイクドロー（攻撃ダメージを受けた点数分ドロー）
5. モンスター攻撃（相手召喚獣を攻撃対象に選べる。場防御と同じ判定式）
6. 新術式 `CMD-PURGE` 追撃粛清（相手の消耗中召喚獣 1 体をスタックごとトラッシュ）
7. デッキ 25 枚 / 同名 2 枚 / power 3+ は 5 枚まで

改訂後の実測値（challenger 同士 750 戦）: 2 点ビハインド逆転 53.3%、先 2 点差側勝率 56.3%、最大スイング 3 点以上 94%、平均 17.0 ターン、先攻勝率 48.7%、デッキ切れ決着 5.5%。

### 1.2 守るべき設計原則（変更前に必ず読むこと）

以下は検証の結果として意図的に選ばれた設計です。変更する場合はシミュレーションで再検証してください。

- **コスト = power の完全一致。** power 4 だけコストを変える案（3 で素出し）は「直感的でない」ため却下済み。基本 3 アクション化がその代替です。
- **手札は逆転の資源、盤面は優勢側の資源。** ブレイクドローで劣勢側の手札が増えるため、手札から撃つ除去（追撃粛清）は逆転装置として、盤面から撃つ除去（モンスター攻撃）は優勢側の決着装置として機能します。両者は互いの欠点（消耗戦化 / 雪だるま化）を相殺するため同時採用しています。
- **検証済みで却下された案（再提案しないこと。再提案するなら新データを添えること）:**
  - 無条件除去の術式 → 逆転率が改善せず試合が延びるだけ（検証済み）
  - 「消耗中の召喚獣への攻撃は必ず破壊」ルール → 消耗戦化のみでリターンなし（検証済み）
  - デッキ 30 枚 → デッキ切れは 2% まで減るがキーカード希釈で逆転率が 44% まで低下（検証済み）
  - power 3+ 上限の 6 枚以上への緩和 → 大型偏重で消化試合度が悪化（検証済み）
  - 序盤限定の召喚コスト割引 → 変則ルールとして却下（3 アクション化で代替）
- **デッキ切れ決着 5〜8% は許容範囲。** コントロール戦略の勝ち筋として意図的に残しています。プレイテストで不評な場合のみデッキ枚数増で緩和します。

### 1.3 検証基盤の使い方

- リーグ検証: `python3 -m ai_break_duel.cli league --games-per-pair 100 --seed <seed> --decks break control fire water wind earth --out tmp/<name>`
  - 必ず 2 シード以上で回して平均を取ること（1 シードだと構成変更の影響とノイズを区別できない）
  - 合格基準: 単色 4 デッキが 45-55%、先攻勝率 48-52%
- 盛り上がり指標: `simulate` の `matches.jsonl` に各手番の `turn_start` イベントとライフ配列が記録されます。ここから「先制ダメージ手番」「リード交代回数」「勝者の最大ビハインド」「最終ライフ差」「決着形態」を集計できます。目安値は `docs/balance-history.md` の 2026-07-03 エントリを参照。
- カード別統計: `simulate` の `summary.json` 内 `card_usage` にカード別の使用・攻撃・防御などのイベント数が集計されます。
- 全体チェック: `npm run check`（typecheck + TS unit + build + Python unittest）。カード効果を追加・変更したら `src/game/cardEffectCoverage.test.ts` への登録が必須（怠ると unit テストが落ちる設計）。
- 実装は Python（`ai_break_duel/`、シミュレータの正）と TypeScript（`src/`、ブラウザ UI）の二重実装です。**ルールやカードを変えたら必ず両方に同じ変更を入れ、両方のテストを更新してください。**
